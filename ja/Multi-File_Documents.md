# <%= @title %>

Marked では、あるファイルを別のファイル内に含めるために、いくつかの異なる構文が使用できます。

## マークされた構文

行の先頭に特別な構文 `<<[path/file]` を使用すると、単一のプレビュー ドキュメントに外部ファイルを含めることができます。行の上下には空白行が必要です。また、パスがスラッシュ (`/`) またはチルダ (`~`) で始まらない限り、パスはメイン文書に対する相対パスであるとみなされます。ファイルへの絶対パスを定義するには、スラッシュ (ルート ディレクトリ) とチルダ (ホーム ディレクトリ) を使用できます。外部ファイルがメインドキュメントと同じフォルダーにある場合、パスは必要ありません。ファイル名 (大文字と小文字が区別され、拡張子も含む) を角括弧内に入力するだけです。

メタデータ ヘッダー「Include Base」または「Transclude Base」を使用して、インクルードされるファイルのベースの場所を変更できます。例:

    ベースをトランスクルード: ~/デスクトップ

*含まれているファイルを含むドキュメントを表示する場合、「I」(shift-i) を入力すると、どの含まれているファイルが表示領域にあるかを確認できます。インクルードされたファイルのパスが表示されているときに Return キーを押すと、インクルードされたファイルがデフォルトのエディターで開きます。*

この機能を使用すると、複数のファイル (章ごとにファイルなど) を使用して大規模なドキュメント/ブックを構築し、単一のインデックス ファイルでドキュメントの順序を指定できます。ファイルの名前やフォルダーの構成は関係ありません。マーク付きで開いたファイルはインデックスとみなされ、その中にリストされているファイルが含まれます。 3 部構成のドキュメントのインデックス ファイルの例:

フォルダー構造:

![][1]

   [1]: 画像/マルチファイルドキュメント構造.jpg @2x 幅=316px 高さ=163px クラス=センター

インデックス.md:

	# ドキュメントのタイトル

	## セクション 1

	<<[セクション/section1.md]

	## セクション 2

	<<[セクション/section2.md]

	## セクション 3

	<<[セクション/section3.md]

Marked で Index.md を開くと、その内容が表示され、含まれている 3 つのファイルがすべて内部で展開されます。含まれているすべてのファイルの変更が監視されます。マーク付きで開いているドキュメントとは異なり、含まれるファイルの追跡は更新を取得するために Spotlight に依存しており、ディスク上の Spotlight インデックス付きフォルダーに存在する必要があります。

[この構文のバリエーション](Special_Syntax.html#includingcode) を使用して、コード スニペットと生の HTML またはテキストを含めることもできます。

インクルードを含むドキュメントの最終的な HTML エクスポートには、インポートされたテキストの先頭と末尾に、インクルードされたファイルの相対パスを含む HTML コメントが含まれます。

**注意:** ドキュメントに含まれるファイルが多いほど、プレビューの全体的なコンパイル時間は遅くなります。 Marked はプロセスの最適化とキャッシュを試みますが、ドキュメントのサイズが大きくなるにつれて、レンダリングの遅延が発生することが予想されます。

## MultiMarkdown トランスクルード構文

新しい MultiMarkdown 仕様に基づいた `{{filename}}` 構文を使用することもできます。 Marked は MMD メタデータ内の `Transclude Base: path` を認識し、それをファイル トランスクルージョンのベースとして使用します。

Transclude Base は親ドキュメント内でのみ認識され、追加のインクルードドキュメント内では認識されません。すべてのネストされたインクルードには、最初のトランスクルード ベース、または親ドキュメントの場所に基づくパスが必要です。

処理せずにファイルをインクルードするために MultiMarkdown が提供するフェンスされたコード構文は、Marked では機能しません。これを行うには、`<<(file)` (コード ブロック) または `<<{file}` (生) 構文を使用してください。

## IA ライター ブロックの構文

マーク付き 2.5.11+ は、IA Writer [Content Block][ia] 構文をサポートします。これは、単独の行のスラッシュ (`/`) で始まる参照です。コード サンプル、画像、マークダウン ファイル、または CSV ファイルを使用できます。すべてはインクルードされたファイルの拡張子に基づいて適切に処理され、可能であれば CSV は [Markdown][csv] テーブルに変換されます。

IA Writer では、インクルードされたファイルは iCloud コンテナに取り込まれ、必ずしも「実際の」パスを必要とするわけではありません。マーク付きでは、インクルードされるファイルがプレビューされているファイルと同じフォルダーに既に存在しない限り、この構文は絶対パスまたは相対パスとともに使用する必要があります。最初のスラッシュは無視されるため、絶対パスの場合は 2 つのスラッシュから始めます。

プレビューされているドキュメントと同じフォルダー内のコード スニペット:

    /スニペット.h

「images」というサブディレクトリへの相対パス:

    /images/image.png "オプションのタイトル"

ドキュメントフォルダーへの絶対パス:

    //ユーザー/ユーザー名/Documents/content.csv

[ia]: https://github.com/iainc/Markdown-Content-Blocks
[csv]: Creating_Tables_using_CSV_files.html

## アウトライン、マインド マップ、CSV インクルードの変換方法

`<<[path]` または IA Writer ブロック構文を使用して特定のファイル タイプを含めると、Marked は生のファイル コンテンツを挿入する代わりにそれらを変換します。アウトラインとマインド マップの動作は、ファイル拡張子と [設定: アプリ → マインド マップ/アウトライン][マインドマップ] の設定によって異なります。 CSV/TSV ファイルは常に Markdown テーブルに変換されます ([CSV ファイルを使用したテーブルの作成][csv] を参照)。

|フォーマット |拡張子 | 「マーメイドとして埋め込む」が**オン**の場合 | **オフ**の場合 |
|----------|-----------|-------------------------------------|--------------|
| **iThoughts X** | .itmz |人魚のマインドマップ図 (整理整頓されたツリー) | .itmz | からのプレビュー画像
| **マインドマネージャー** | .mmap |人魚のマインドマップ図 |入れ子になったマークダウン リスト |
| **フリーマインド** | .mm |人魚のマインドマップ図 |入れ子になったマークダウン リスト |
| **OPML** | .opml |人魚のマインドマップ図 |入れ子になったマークダウン リスト |
| **オムニアウトライナー** | .o概要 |人魚のマインドマップ図 |入れ子になったマークダウン リスト |
| **自転車** | .バイク |マーメイド マインドマップ (ルート ノードとしてのファイル名) |ネストされたマークダウン リスト (ドキュメント タイトルなし) |
| **CSV / TSV** | .csv、.tsv |マークダウン テーブル ||
| **RTF / RTFD** | .rtf、.rtfd | || RTF 変換によるマークダウン ([RTF および RTFD のサポート](RTF_Support.html) を参照)
| **PDF** | .pdf | || メインドキュメントとして開いた場合の PDF 変換によるマークダウン ([PDF サポート](PDF_Support.html) を参照)

各アウトライン/マインド マップ形式には、*マインド マップ/アウトライン* (マインド マップ、OPML ファイル、バイク アウトライン、OmniOutliner アウトライン) の下に独自のチェックボックスがあります。フォーマットをオフに切り替えると、そのタイプに対してのみ「オフ」動作が使用されます。形式の詳細については [マインド マップとアウトラインの埋め込み](Embedding_Mind_Maps_and_Outlines.html) を参照し、これらのオプションを変更するには [設定: アプリ][マインドマップ] を参照してください。 CSV変換の詳細については、「CSVファイルを使用したテーブルの作成」【csv】を参照してください。

## 書籍のフォーマット

Marked は、[Leanpub][lp]、[GitBook][gb]、mmd\_merge (MultiMarkdown) などの形式のインデックス ファイルもサポートしています。ブック形式のインデックスに含まれるファイルの変更が監視され、その結果、上記の「Index.md」の例と同様に、コンパイルされたドキュメントの完全なプレビューが表示されます。

### リーンパブ

Leanpub/GitBook サポートの下の {% prefspane Apps %} のオプションを有効にすると、「Book.txt」という名前のファイルが自動的に Leanpub インデックス ファイルとして扱われます。古い「frontmatter:」形式も認識されます。

[Leanpub ドキュメント][lpdocs]

Leanpub Book.txt の例:

    前題:
    謝辞.txt
    まえがき.txt
    はじめに.txt
    主要事項:
    マークダウン.txt
    サンプルブック.txt
    画像の挿入.txt


### mmd_merge

mmd\_merge の場合、Marked では最初の行が「#merge」である必要があります (mmd\_merge の特別な Marked トリガー。コメントとして扱われ、他のプロセッサによって無視されます)。

[mmd_merge ドキュメント。][mmdm]

mmd_merge の例:

    #マージ
    メタデータ.txt
    第1章.md
        サブチャプタ-1-1.md
        サブチャプター 1-2.md
    第2章.md
        サブチャプター 2-1.md
        サブチャプター 2-2.md
    FAQ.md
    謝辞.md

### GitBook

GitBook の書式設定では、Markdown リストを使用して構造と目次を作成します。 Leanpub/GitBook サポートの {% prefspane Apps %} で GitBook サポートが有効になっている場合、SUMMARY.md という名前のファイルが読み込まれ、自動的に mmd_merge 形式に変換され、GitBook ドキュメントの完全なプレビューが可能になります。

[GitBook ドキュメント][gbdocs]

GitBook SUMMARY.md の例:

    # まとめ

    * [パート 1](part1/README.md)
        * [書くのはいい](part1/writing.md)
        * [GitBookはいいよ](part1/gitbook.md)
    * [パート II](part2/README.md)
        * [フィードバックは大歓迎です](part2/フィードバック_プリーズ.md)
        * [作成者のためのより良いツール](part2/better_tools.md)

GitBook では SUMMARY.md の目次でアンカーを使用できますが、Marked はこれらを無視し、基本ドキュメントを 1 回だけ含めます。

## 複数ファイルのドキュメント プレビュー機能

![含まれるファイルの境界][2]

   [2]:images/includeboundaries.png @2x width=859px height=239px class=center

含まれているファイルを含むドキュメントを表示する場合、2 つの機能を使用して、どのファイルを見ているのかを把握できます。

* **キーボード:** {% kbd shift I %} を押すと、プレビューのスクロール位置に現在表示されているファイルのタイトルを示すポップアップが表示されます。
    ※{% kbd I %}に続いて{% kbd Return %}を押すと、表示されているファイルを外部エディタで編集できます。
* **マウス:** 歯車メニュー ({% kbd ctrl cmd B %}) から「含まれるファイルの境界を表示」を選択すると、プレビューの左側に色付きのバーが追加され、含まれるファイルの先頭と末尾を示すようにセグメント化されます。ネストされたインクルードも表示されます。このバーのセクションにマウスを置くと、それが表すファイルの名前が表示され、それをクリックすると、選択したエディタでそのファイルが開きます。
