## コールアウト

## クマ/黒曜石 ##

Marked は、Obsidian と Bear で使用される構文 (特別にフォーマットされたブロック引用符) を使用したコールアウトをサポートします。

	> [!NOTE] ノートのタイトル
	> 追加テキスト

`[!NOTE]` の「注意」は大文字と小文字が区別されません。次のいずれかになります。

- 注意
- 要約、概要、TLDR
- 情報
- TODO
- ヒント、ヒント、重要
- 成功、確認、完了
- 質問、ヘルプ、よくある質問
- 警告、注意、注意
- 失敗、失敗、欠落
- 危険、エラー
- バグ
- 例
- 引用、引用

これらは特別にフォーマットされたブロックを作成します。

`+` または `-` を使用して、吹き出しを折りたたむことができます。プラス記号 (`+`) は、吹き出しは折りたたむことができますが、デフォルトでは開くことを意味します。マイナス記号 (`-`) は、吹き出しは折りたたみ可能ですが、デフォルトでは閉じていることを意味します。

### Xcode プレイグラウンド ###

Xcode Playground ファイルをプレビューする場合、Marked はネイティブの Xcode Playground コールアウト構文をサポートします。

	- 注意: タイトルは任意です
	吹き出しの内容がここに入ります。

コールアウト タイプ (「注意」など) は大文字と小文字が区別されず、次の Xcode Playground コールアウト タイプのいずれかを使用できます。

- **注意** - 情報吹き出しとしてスタイル設定されています
- **作成者** - メタデータ コールアウト
- **作成者** - メタデータ コールアウト
- **バグ** - エラー/危険のコールアウト
- **複雑さ** - メモ形式の吹き出し
- **著作権** - メタデータのコールアウト
- **カスタム コールアウト** - サンプル スタイルのコールアウト
- **日付** - メタデータ コールアウト
- **例** - コールアウトの例
- **実験** - ヒント形式の吹き出し
- **重要** - 情報スタイルのコールアウト
- **不変** - メモ形式の吹き出し
- **注** - 注の吹き出し
- **前提条件** - メモ形式のコールアウト
- **事後条件** - 注記スタイルのコールアウト
- **注釈** - 注記スタイルの吹き出し
- **必須** - メモ形式の吹き出し
- **関連項目** - 情報スタイルのコールアウト
- **以降** - メタデータ コールアウト
- **バージョン** - メタデータ コールアウト
- **警告** - 警告コールアウト

コロンの後のオプションのタイトルは、吹き出しのタイトルとして使用されます。タイトルが指定されていない場合は、吹き出しタイプの名前がタイトルとして使用されます。

吹き出しの内容は次の行にすぐ続きます (空白行は必要ありません)。コンテンツは、空行、次の吹き出し、囲われたコード ブロック、またはドキュメントの終わりまで続きます。

例:

````値下げ
- 重要: パフォーマンスに関する注意事項
このアルゴリズムの複雑さは O(n log n) です。

- 例: クイックソート
実装方法は次のとおりです。

```swift
func quickSort(_ array: [Int]) -> [Int] {
   // implementation
}
```
````

タイトルを完全に省略することもできます。

	- 警告
	これはカスタム タイトルのない警告です。

これらの吹き出しは、Marked の吹き出し形式に自動的に変換され、適切にスタイル設定されます。元の吹き出しタイプは `data-callout` 属性に保持され、必要に応じてカスタム CSS スタイルを使用できます。

> この機能は、Xcode Playground ファイルをプレビューする場合にのみ機能します (`.playground`)。通常のマークダウン ファイルでは、この構文は処理されません。


## 目次

`<!--TOC-->` を使用して、文書内の目次を表示する場所を指定できます。これが設定されている場合、環境設定のオプションが上書きされ、保存時や印刷時だけでなく、プレビュー ウィンドウにも常に表示されます。コンテンツ内に複数の `<!--TOC-->` 指定子がある場合でも、目次は 1 回だけ表示されます。

上記のタグに `max#` を追加すると (# は 1 ～ 5 の数字)、表示される目次の深さを制御します。たとえば、`<!--TOC max2-->` はリストの第 1 レベルと第 2 レベルのヘッダーのみを表示します。 `<!--TOC min2-->` で最小ヘッダーレベルを指定することもできます。最大値と最小値は、ネスト レベルではなく、実際の見出しレベルに基づいています。最大値と最小値は併用できます。

Marked は、MultiMarkdown スタイルの `{{TOC}}` と Pandoc スタイルの `{{TOC:2-6}}` も認識します。ここで、`2-6` は、含めるヘッダーの最小レベルと最大レベルの範囲です。

デフォルトでは、{% prefspane Export %} で「目次の印刷」が有効になっている場合、目次はドキュメントの最初のページに印刷されます。ドキュメント内にマーカーが存在する場合は、代わりにその位置にマーカーが配置されます。

I> {% prefspane Export %} で、ネストされた目次階層の各レベルの番号付けまたは文字の種類を指定できます。

## 改ページ

次の構文を使用して、印刷/PDF 出力の改ページを強制できます。

```html
<!--BREAK-->
```

トークンを単独で行に配置すると、その時点で強制的に新しいページを作成するマークアップが生成されます。 Marked は Leanpub フォーマットも理解しています。

	{::改ページ /}

## 自動スクロールが一時停止します [一時停止]

マークは、[自動スクロール](Autoscroll.html) 機能を使用してテレプロンプターとして機能できます ([テレプロンプター スタイル](https://markedapp.com/styles/preview?style=teleprompter) を追加する必要があります)。その際、文書内に一時停止を含めると便利な場合があります。以下を使用してこれを実行します。

```html
<!--PAUSE:X-->
```

ここで、`X` は Marked が一時停止する秒数です。したがって、`<!--PAUSE:15-->` を挿入すると、文書内のその点が画面の中央に達したときに 15 秒間の一時停止が生じます。

## ファイルに含まれるもの

追加ファイルの内容は、次の構文を使用して挿入できます。

	<<[フォルダー/ファイル名]

ファイルへのパスは、インデックス ファイルに対して相対パスまたは絶対パスにすることができます。インクルードはネストできます。これと同じ構文をインクルードされたファイル内で使用できます。相対パスを使用している場合、ネストされたファイル内のインクルードはそのファイルを基準とする必要があります。 ***ただし、*** MultiMarkdown は最初に開かれたファイルの場所に基づいてすべてを処理するため、すべての画像パスまたはその他の埋め込みは、子ドキュメント内に存在する場合でも、最初の親ファイルを基準とする必要があります。

MultiMarkdown メタデータと YAML ヘッダーは、レンダリング前にインクルードされたファイルから自動的に削除されます。これにより、ヘッダーがドキュメントに表示されなくなりますが、「基本ヘッダー レベル」などのメタデータは、組み込まれたドキュメントでは無視されることに注意してください。

T> 含まれているファイルを含むドキュメントを表示する場合、「I」(shift-i) を入力すると、どの含まれているファイルが表示領域にあるかを確認できることに注意してください。

詳細については、["複数ファイル ドキュメント"][ext] を参照してください。

## コードを含める

マークすると、上記のファイル インクルードと同様の構文を使用して、外部ファイルをコードとしてインクルードできます。

	<<(フォルダー/ファイル名)

角括弧の代わりに括弧を使用していることに注意してください。 Leanpub 構文との互換性を保つために、Marked はタイトルを含む前述の角括弧のセットも認識しますが、現時点では Marked では何も行われません。

	<<[コードタイトル](フォルダ/ファイル名)

指定したファイルの内容はドキュメントの pre>code ブロック内に挿入され、自動構文ハイライトが有効になっている場合は使用できるようになります。コード ブロックはネストできず、MultiMarkdown では処理されません。カスタム プロセッサは、作成された pre>code ブロック上で引き続き実行されます。

## 未処理のテキストまたは HTML を含む

E> **注:** この機能は上級ユーザー向けです。

MultiMarkdown (またはカスタム プロセッサ) で処理すべきではない生の HTML またはその他のテキストを含める場合は、中括弧 (`{}`) を使用して、ドキュメントの残りの部分を処理した後、ファイルを含めることができます。

	<<{フォルダ/raw_file.html}

これらのファイル内ではインクルード構文は認識されず (ネストなし)、ファイルの **生** コンテンツが最終的な HTML 出力に挿入されます。これは、テキスト プロセッサを停止させたり、変換したくないときに変換/エスケープしたりすることなく HTML を挿入する場合に最適ですが、挿入した内容のドキュメントの書式設定を確実に保持するための安全策がほとんどないので注意してください。
